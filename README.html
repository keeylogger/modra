⚡ The declarative full-stack language

One file.
A whole web app.

Modra unifies UI, state, server endpoints, and database schema in a single declarative syntax — then compiles to a real React frontend, a real Node backend, and real Postgres tables. No glue code. No API client. No drift.

Get started → ▶ Live playground vs other languages
🧠

One mental model

UI, state, side-effects, server endpoints, and SQL tables share one syntax. Same operators, same scoping, same identifiers.

⚡

Reactive by default

The <- operator binds a value reactively. Change the source, every dependent re-renders or re-runs. No hooks ceremony.

🌉

Auto-bridged

Call a @@target: Server endpoint from the client like a local function. The compiler generates the fetch glue and types.

🗃️

Database, first-class

Declare a Database with typed Tables — get Postgres DDL, a typed query API, and migrations for free.

🎨

Plain UI attributes

Style with ::Size.Medium, effects with <:Hover. No CSS-in-JS noise, no className gymnastics.

🚪

Native escape hatches

Need something Modra can't express? Drop into Native<Python> { … } or Native<TypeScript>. Typed in, typed out.

Hello, world

A reactive counter — full component, runnable as written:

modra
Number: Count <- 0

Action: Increment -> ( Count <- Count + 1 )
Action: Reset     -> ( Count <- 0 )

Component: Counter -> (
  Text: Display <- "Count is " + Count

  form (
    Submit  label: "Increment"  Click -> Increment
    Submit  label: "Reset"      Click -> Reset
  )
)

That compiles to:

  • a React component with a useState hook and JSX,
  • … nothing else, because there are no server or database bits yet.

Add a backend (still one file)

modra
using Backend.Database as DB

Database: Postgres -> (
  Table: Visits -> (
    String:   ID @Primary
    DateTime: At
    String:   UserName
  )
)

Endpoint: Greet(name: String) -> (
  Record: row <- DB.Visits.Insert(ID: UUID(), At: Now(), UserName: name)
  Return: "Hello, " + name + "!"
)

Component: Hello -> (
  form (
    InputField::Name      placeholder: "Your name"
    Submit                label: "Say hi"  Click -> Greet(Name)
  )
)

The compiler produces:

  • a React page with a controlled input and an auto-generated greet() client that POSTs to /api/greet,
  • a Node + Express handler that calls db.visits.insert(...),
  • a Postgres DDL for the visits table,
  • a shared RPC type file keeping client and server in sync.
✓

Try the registration example in the live playground — edit on the left, see emitted code on the right, in real time.

Why Modra exists

Modern web stacks scatter one feature across five files: a React component, a hook, an API route handler, a database migration, and a typed client wrapper. They drift. They duplicate. Modra collapses them.

ConcernTypical stackModra
UI markupJSX / Svelte / Vue templateform (...) tree
UI stateuseState, signals, storesNumber: X <- 0
Server endpointExpress route + Zod schemaEndpoint: X -> ( ... )
Client callHand-written fetch + typesJust call the endpoint
Database schemaSQL file or ORM modelTable: T -> ( ... )
ValidationZod / Yup / JoiTypes are the schema

Installation

Modra ships as an npm package. You need Node 20+ and npm 10+.

npm

bash
npm install -g modra
# or, per project:
npm install --save-dev modra

Verify

bash
modra --version
# modra 1.0.0

modra --help
# Usage: modra <command> [options]
# Commands: init, build, dev, check, parse, lex

Editor setup

  • VS Code: a syntax-highlighting grammar ships with the package. Open a .modra file and pick "Modra" from the language picker.
  • Diagnostics: Modra's CLI prints Rust-style errors with source carets. Pipe through your normal terminal.
  • Format on save: coming in 1.1 (modra fmt).

From source

bash
git clone https://github.com/keeylogger/modra.git
cd modra
npm install
npm test          # 294 tests
npm run build     # build the compiler
npm link          # expose the `modra` binary
modra init my-app
ℹ

Modra is currently alpha. The CLI and emitted code are stable enough for prototypes; pin a version and expect a few rough edges.

Quick start

Go from npm install to a running full-stack app in under five minutes.

1 · Scaffold a project

bash
modra init taskboard
cd taskboard

You'll get a folder with:

tree
taskboard/
├── taskboard.modra   # your source
├── package.json
└── README.md

The starter taskboard.modra looks like this:

modra
Number: Count <- 0

Action: Increment -> ( Count <- Count + 1 )
Action: Reset     -> ( Count <- 0 )

Component: TaskBoard -> (
  Text: Status <- "Tap count: " + Count

  form (
    Submit  label: "Tap"    Click -> Increment
    Submit  label: "Reset"  Click -> Reset
  )
)

2 · Build

bash
modra build taskboard.modra -o ./dist

Modra emits a complete project tree:

tree
dist/
├── package.json        # Vite + React + Express setup
├── vite.config.ts
├── tsconfig.json
├── .env.example
├── README.md
├── src/                # React frontend
│   ├── main.tsx
│   ├── App.tsx
│   ├── components/TaskBoard.tsx
│   └── runtime/        # client stdlib (Hash, Toast, Navigate, …)
├── server/             # Node + Express backend
│   ├── index.ts
│   ├── db.ts
│   ├── routes/
│   └── runtime/        # server stdlib (AuthToken, Hash, …)
├── shared/             # auto-generated bridge
│   ├── rpc.ts
│   └── types.ts
└── schema.sql          # Postgres DDL

3 · Run

bash
cd dist
npm install
npm run dev   # client on :5173, server on :3000

4 · Develop with live rebuilds

From your .modra source directory, run:

bash
modra dev taskboard.modra -o ./dist

The Modra compiler watches the source file and regenerates the project on every save. Pair it with npm run dev inside ./dist for end-to-end hot reload.

5 · Add a server endpoint

Open taskboard.modra, add a server endpoint, and call it from a button. The compiler will wire up the API.

modra
Number: Count <- 0

Action: Increment -> ( Count <- Count + 1 )

Endpoint: Save(count: Number) -> (
  Log: "Saving count " + count
  Return: true
)

Component: TaskBoard -> (
  Text: Status <- "Count: " + Count

  form (
    Submit  label: "+"     Click -> Increment
    Submit  label: "Save"  Click -> Save(Count)
  )
)
✓

Save the file. modra dev rebuilds, the React UI hot-reloads, and clicking Save hits /api/save on your Node backend.

Modra's grammar is small but expressive. Most of the language fits on a single reference card.

The four assignment operators

This is the heart of Modra. Read them aloud — the symbols match the dataflow direction.

OpReads asMeaning
:"is"Static assignment / declaration label
<-"reactively becomes"Reactive binding (re-evaluates when dependencies change)
->"then"Event handler / endpoint body / arrow
<->"is synced with"Two-way binding
<:"apply effect"Attach an effect / decorator to a value
modra
// Static literal — value never changes after init.
String: AppName <- "Modra"

// Reactive — re-runs whenever Count changes.
Number: Count <- 0
String: Label <- "Count is " + Count

Action: Tap -> ( Count <- Count + 1 )

Component: Demo -> (
  // Event handler — Click fires the named Action.
  Submit  label: "Tap"  Click -> Tap

  // Two-way binding — Element :: state.
  InputField::Email

  // Apply effect — declarative side-effects on a value.
  Avatar: Pic
  Pic <: HoverEffect(pop)
)

The selector operator ::

Use :: to bind an element to a target name (two-way), or to pick a variant on a value.

modra
Component: Demo -> (
  form (
    // Two-way bind: InputField <-> Email, plus a placeholder attr
    InputField::Email     placeholder: "you@domain.com"
    InputField::Password  placeholder: "•••••••"
  )

  // Variant selector — applies a kind/size to an element.
  Avatar: Hero (size: Large)
  Button: Buy  (color: Primary, label: "Buy now")
)

Reactive types

A reactive value still has a static type — Modra just adds an implicit "subscribable" wrapper at compile time.

modra
Number: A <- 1
Number: B <- A * 2   // B is reactive: when A changes, B re-evaluates.

State & reactivity

In Modra, data flows downhill. Declare a state value with <-, and every expression that reads it stays in sync.

Local state

modra
String: Name <- "Ada"
Number: Age  <- 36
String: Bio  <- Name + " is " + Age + " years old."

Component: Profile -> (
  form (
    InputField::Name
    InputField::Age
    Text: BioLine (Bio)
  )
)

Watch what happens:

  • InputField::Name sets up an automatic two-way binding to Name.
  • Bio is reactive — it reads Name and Age, so it re-renders whenever either changes.
  • No useState, no useEffect, no subscribe call.

Derived values

modra
Number: Quantity  <- 1
Number: UnitPrice <- 9.99
Number: Subtotal  <- Quantity * UnitPrice
Number: Tax       <- Subtotal * 0.08
Number: Total     <- Subtotal + Tax

This is a one-way dependency graph; the compiler topologically sorts updates.

Sync (two-way)

For symbolic mirrors, <-> keeps two state values aligned both ways:

modra
String: Query <- ""
String: Mirror <- ""

Component: Search -> (
  Query <-> Mirror

  form (
    InputField::Query  placeholder: "Search…"
    Text: T (Mirror)
  )
)

Implicit state from bindings

You can skip the declaration entirely. The compiler creates the state for you:

modra
Component: Login -> (
  form (
    InputField::Email      // auto-declares String: Email <- ""
    InputField::Password   // auto-declares String: Password <- ""
    Submit  label: "Sign in"  Click -> Authenticate(Email, Password)
  )
)

Declaring an action

Endpoints

An Endpoint is a server-side function that the compiler auto-exposes as a typed API and an auto-generated client.

The bridge in one picture

modra
String: Reply <- ""

Endpoint: Greet(name: String) -> (
  Return: "Hello, " + name
)

Action: SayHi -> (
  // Looks like a local call. The compiler emits a fetch.
  Reply <- Greet(Name)
)

Component: Hello -> (
  form (
    InputField::Name
    Submit  label: "Say hi"  Click -> SayHi
    Text: Out (Reply)
  )
)

The compiler emits:

  • Server: Express handler POST /api/greet
  • Shared: shared/types.ts with GreetRequest / GreetResponse
  • Client: shared/rpc.ts exposing greet(name): Promise<string>
  • UI: a Button that awaits the call and sets Reply

Validation comes for free

The endpoint's parameter types are checked at the wire boundary:

modra
Endpoint: CreateOrder(items: Array<Object>) -> (
  if items.length == 0 -> (
    Throw: Error("Cart is empty")
  )
  String: orderId <- UUID()
  Record: row <- DB.Orders.Insert(ID: orderId, Items: items)
  Return: orderId
)

Authentication

modra
Endpoint: Me -> (
  String: token <- AuthToken()
  Record: u <- DB.Users.Find(token)
  Return: u
)

Returning errors

modra
Endpoint: Login(email: String, password: String) -> (
  Record: user <- DB.Users.Find(email)
  if user == None -> ( Throw: Error("Unknown email") )
  if user.PasswordHash != Hash(password) -> ( Throw: Error("Bad credentials") )
  Return: IssueToken(user.ID)
)

Database

A Database block compiles to Postgres DDL and a typed query API. No ORM. No migration tool. No drift.

Declaring tables

modra
using Backend.Database as DB

Database: Postgres -> (
  Table: Users -> (
    String:   ID @Primary
    String:   Email @Unique
    String:   PasswordHash
    String:   Role
    DateTime: CreatedAt <- Now()
  )

  Table: Posts -> (
    String:   ID @Primary
    String:   Title @Indexed
    String:   Body
    String:   AuthorID @References(Users.ID)
  )
)

Emits:

sql
CREATE TABLE users (
  id             VARCHAR  PRIMARY KEY,
  email          VARCHAR  UNIQUE NOT NULL,
  password_hash  VARCHAR  NOT NULL,
  role           VARCHAR  NOT NULL,
  created_at     TIMESTAMP NOT NULL DEFAULT NOW()
);

CREATE TABLE posts (
  id         VARCHAR PRIMARY KEY,
  title      VARCHAR NOT NULL,
  body       VARCHAR NOT NULL,
  author_id  VARCHAR NOT NULL REFERENCES users(id)
);
CREATE INDEX idx_posts_title ON posts(title);

Querying

modra
Endpoint: ReadStuff(someId: String) -> (

Native bridges

When Modra can't express something — say, calling NumPy or a specific TypeScript library — drop into a typed native block.

Calling Python

modra
Endpoint: Sentiment(text: String) -> (
  Native<Python>(in: text; out: label) {
    from transformers import pipeline
    clf = pipeline("sentiment-analysis")
    label = clf(text)[0]["label"]
  }
  Return: label
)

Calling TypeScript

modra
Action: SlugifyTitle(title: String) -> (
  Native<TypeScript>(in: title; out: slug) {
    slug = title.toLowerCase().replace(/[^a-z0-9]+/g, "-");
  }
  Return: slug
)

How it works

  • Each native block declares typed inputs on the left and typed outputs on the right.
  • The compiler emits a stub file (natives/<Name>.py, natives/<Name>.ts) and a runtime helper that calls it.
  • You implement the body once. Modra never reads or modifies the stub.

Directives

Directives starting with @@ control how the compiler treats a file or a specific declaration.

Common directives

DirectiveScopeEffect
@@target: Clientfile or declForce client emission
@@target: Serverfile or declForce server emission
@@target: Sharedfile or declEmit on both sides
@@unsafedeclSkip auth checks on this endpoint (use sparingly)
@@deprecated: "msg"declEmit a warning when used
@@structure: ProceduralfileLock paradigm (1.1)

File-level vs declaration-level

modra
@@target: Client       // file default

Component: Form -> (   // → client
  Text: T ("hi")
)

Endpoint: Save -> (    // intrinsically server
  Return: true
)

@@target: Server
Action: Audit -> (     // server, even though the file default is client
  Log: "audited"
)

For 1.0, diagnostics cannot be suppressed line-by-line. The recommended pattern is to fix the cause or wrap the unknown logic in a native bridge:

Where Modra is not the right tool

  • Heavy compute / numeric kernels — use Python or Rust and call it via Native<…>.
  • Low-latency systems (HFT, kernels, embedded) — Modra targets web stacks only.
  • Large existing React codebases — Modra is greenfield-first today.
  • Mobile-native apps — emitting React Native is on the 1.2 roadmap.

// 08 · Native bridges

FAQ

Is Modra a framework or a language?

A language. It has its own lexer, parser, type system, and compiler. It happens to emit React + Node, but those are targets, not deps.

Can I drop Modra into an existing app?

Not directly in 1.0 — Modra emits its own project skeleton. You can copy out the emitted components and integrate them piecemeal. First-class "single-component" emission is on the 1.2 roadmap.

How fast is the compiler?

Hand-rolled, zero deps, ~6 ms for a typical file on a modern laptop. The playground recompiles on every keystroke.

Does it produce idiomatic React / Node?

Yes — the emitted code is hand-readable. Components use hooks, the backend uses Express, the SQL is plain DDL. No runtime magic.

What about TypeScript types in the emitted code?

The bridge (shared/rpc.ts, shared/types.ts) is fully typed. Components currently use loose types; full strict TS for the UI tree is on the 1.1 roadmap.

How do I use my own React components?

Wrap them with a Native<TypeScript> block, or register them via using Frontend.React (1.1).

MIT. Use it however you like.
